# 《真实产品案例补充 v0.1》

## 目的
在《槽位在 agent 产品中的应用地图 v0.1》基础上，补充一组真实产品或真实产品类型案例，用来说明：槽位机制在不同 agent 场景中如何出现、位于系统哪一层、以及产品究竟更像参数工具还是更像 slot-based agent。

> 注：这里的案例不要求都与 Fuli Plus 属于同一行业。重点不是行业相似，而是方法相似——是否使用了结构化中间表示层来承接输入、状态、反馈与决策。

## 案例 1：Midjourney
### 产品类型
图像生成工具

### 槽位机制
- 主要以 prompt + 参数命令的形式出现
- 包括 aspect ratio、stylize、chaos、version 等全局控制参数
- 用户常通过风格词、媒介词、构图词堆叠来隐式构造“槽位”

### 槽位位于哪一层
- 层 1：输入表单层（文本提示）
- 层 2：生成控制层（参数控制）

### 特点
- 可生成高质量图像
- 具有较强的 prompt craft 属性
- 用户需要自己承担大量中间表示工作

### 局限
- 槽位未显式结构化
- 用户反馈难进入系统状态
- 更像 prompt-first tool，而不是 slot-based agent

---

## 案例 2：Adobe Firefly
### 产品类型
设计场景中的 AI 生成工具

### 槽位机制
- 通过 style、content type、lighting、color tone、camera angle 等属性面板提供结构化控制
- 某些工作流中支持参考图、风格强度、局部编辑等

### 槽位位于哪一层
- 层 1：输入表单层
- 层 2：生成控制层
- 部分工作流接近层 3：状态表示层

### 特点
- 比纯 prompt 更结构化
- 更贴近设计师常用控制语言
- 降低了 prompt 工程门槛

### 局限
- 用户不满意时，更多还是重新生成或调参数
- 很少把反馈解释为“当前问题来自哪个维度”
- 状态收敛能力有限

---

## 案例 3：Canva / 模板型 AI 设计工具
### 产品类型
模板与品牌内容生成工具

### 槽位机制
- 借由模板字段、品牌色、风格项、版式选择、目标平台等结构化字段来承接用户输入
- 一些品牌系统会把 logo、主色、副色、语气等沉淀为 profile 槽位

### 槽位位于哪一层
- 层 1：输入表单层
- 层 2：生成控制层
- 层 5：长期资产层（品牌 profile）

### 特点
- 槽位非常实用，尤其适合重复生成
- 长期资产属性较强
- 通过 schema 化信息提高一致性

### 局限
- 反馈闭环通常较弱
- 系统不一定根据用户每轮偏好自动更新内部设计逻辑
- 更偏向资产驱动工具，而非强 agent 闭环

---

## 案例 4：Krea / 实时探索型生成工具
### 产品类型
实时图像探索与操控工具

### 槽位机制
- 通过实时 slider、参考图、局部改动、风格控制等方式提供中间控制层
- 用户通过“看着变”来不断修正结果，而不是完全依赖文本 prompt

### 槽位位于哪一层
- 层 2：生成控制层
- 部分接近层 3：状态表示层

### 特点
- 强探索感
- 用户更容易形成“连续调试”的体验
- 中间控制层比 prompt-first 更直观

### 局限
- 多数判断仍由用户自己完成
- 系统较少主动解释哪一个维度是当前主因
- 更像 exploration tool，而不是 feedback-interpreting agent

---

## 案例 5：旅行规划 / 预订型 agent（广义产品类型）
### 产品类型
需求收敛与规划类 assistant

### 槽位机制
- 系统主动补齐预算、时间、地点、同行人、偏好、约束等字段
- 缺失槽位时会反问用户

### 槽位位于哪一层
- 层 1：输入收集层
- 层 3：状态表示层
- 部分产品达到层 5：长期偏好资产层

### 特点
- 槽位直接服务于规划与推荐
- agent 可以围绕缺失槽位发起追问
- 适合展示“需求收敛型槽位”的典型逻辑

### 局限
- 通常目标相对明确
- 对开放式创意偏好的处理弱于创意系统

---

## 案例 6：语言学习 / 辅导 agent（广义产品类型）
### 产品类型
教育与学习支持系统

### 槽位机制
- 把用户表达映射到语法点、词汇类别、错误类型、熟练度、学习阶段等知识槽位
- 系统基于这些槽位给出解释、练习或复习建议

### 槽位位于哪一层
- 层 2：控制层（决定后续教学内容）
- 层 3：状态表示层
- 层 5：长期学习画像层

### 特点
- 非常典型的“知识映射型槽位”
- 槽位是个性化辅导的基础
- 可积累成长轨迹

### 局限
- 若槽位体系过粗，会造成学习反馈失真
- 若系统只做标签记录、不做动态教学策略更新，就不够 agentic

---

## 案例 7：Journaling / Reflective Agent（广义产品类型）
### 产品类型
记忆整理与反思引导系统

### 槽位机制
- 将用户历史输入压成事件、情绪、关系、主题、未完成事项、长期目标等结构
- 后续基于这些结构生成总结、提问和反思提示

### 槽位位于哪一层
- 层 3：状态表示层
- 层 5：长期资产层

### 特点
- 槽位帮助记忆从“原文堆积”变成“可调用状态”
- 适合支持长期陪伴与个体连续性

### 局限
- 若缺少高质量更新机制，记忆容易变成死档案
- 若没有反馈闭环，难以真正帮助用户形成更深反思路径

---

## 案例 8：Coding Agent / Workflow Agent（广义产品类型）
### 产品类型
任务规划与执行系统

### 槽位机制
- 系统维护目标、约束、子任务、文件上下文、工具可用性、错误状态、下一步动作等工作状态
- 某些系统还会通过 plan / checklist / task state 明确表示执行路径

### 槽位位于哪一层
- 层 3：状态表示层
- 层 4：反馈解释层（错误归因）

### 特点
- 槽位更多服务于系统内部 working memory
- 是 agent 执行稳定性的关键

### 局限
- 这类槽位通常不直接面向用户表达
- 更偏控制状态，而非用户语义建模

## 综合观察
### 1. 很多真实产品都在使用“槽位”，只是名称不同
它们可能叫字段、属性、profile、style controls、plan state、memory schema，但本质上都在承担中间表示层的作用。

### 2. 真正区分工具与 agent 的，不是有没有槽位，而是槽位是否进入闭环
如果槽位只用于一次性输入或控制，它更像工具；如果槽位会跨轮保留、解释反馈、更新状态并影响下一步决策，它才更接近 slot-based agent。

### 3. Fuli Plus 的特殊位置
Fuli Plus 最接近“生成控制型槽位”与“反馈归因型槽位”的交叉点。它的挑战不在于发明字段，而在于让槽位同时承担：
- 用户模糊审美的承接
- 多轮偏好反馈的解释
- 设计状态的持续收敛
